# Requirements To MD

这个 skill 用来把原始需求转成可执行方案，并真正落成 `.md` 文件，而不是只在对话里给一个临时答案。

## 工作方式

1. 先理解需求。
   抽出背景、目标、范围、限制、依赖和交付物。如果用户没说全，不停在追问，先补合理假设并标记 `待确认`。
2. 判断文档类型。
   如果用户明确要“技术方案”“产品方案”“实施计划”“需求分析”，就按用户类型写；没明确时默认写成通用实施方案。
3. 组织方案内容。
   默认按照 [references/solution-template.md](references/solution-template.md) 的结构输出，需要时增减章节。
4. 直接生成 Markdown 文件。
   如果用户指定路径或文件名，按指定位置写入。
   如果用户没指定，默认写入仓库根目录下的 `docs/`，文件名使用 `YYYY-MM-DD-<topic>.md`。
5. 回答里说明结果。
   简要说明文档写到了哪里，列出核心假设、关键风险和下一步。

## 输出要求

- 文档必须是可落地方案，不是泛泛而谈的概述
- 方案里要明确边界：做什么，不做什么
- 如果存在多种方案，至少给出推荐方案和放弃其他方案的理由
- 如果信息不足，仍然输出一版可讨论文档，并把缺口收敛到 `待确认事项`
- 文档语言默认跟随用户；用户用中文时输出中文

## 默认章节

- 背景与问题
- 目标
- 范围
- 约束与假设
- 方案设计
- 实施计划
- 风险与待确认
- 验收标准

## 避免

- 只复述需求，不形成决策
- 因为信息不全就拒绝产出文档
- 把风险、依赖、验收标准漏掉
- 没有落文件，只在对话里返回内容
