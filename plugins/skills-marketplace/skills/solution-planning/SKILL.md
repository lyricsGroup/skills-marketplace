# Solution Planning

这个 skill 用来把需求转成可执行方案，并真正落成 `.md` 文件，而不是只在对话里给一个临时答案。

## 工作方式

1. 先理解需求和约束。
   抽出背景、目标、范围、限制、依赖和交付物。不完整时先补合理假设并标记 `待确认`。
2. 判断是新建方案还是修改方案。
   如果已有文档，优先基于现有文档增量修改。
3. 组织内容并落文件。
   默认按照 [references/solution-template.md](references/solution-template.md) 输出。
4. 如果用户没指定文件名或路径。
   默认写入仓库根目录 `docs/`，文件名使用 `YYYY-MM-DD-<topic>.md`。

## 输出要求

- 明确做什么和不做什么
- 至少给出推荐方案和理由
- 信息不足时仍输出一版可讨论文档

## 给维护者的提示

- 模板结构优先放在 `references/`
- 如果团队文档格式固定，优先改模板，不要把正文说明写得过长
