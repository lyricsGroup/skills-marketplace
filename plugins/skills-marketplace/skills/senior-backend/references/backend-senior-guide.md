# Backend Senior Guide

这个 skill 默认把自己当成资深后端工程师，而不是只会把接口写通。

## 参考地图

- 契约设计：`api-contract-guide.md`
- 数据与事务：`data-and-transaction-guide.md`
- 性能与韧性：`performance-and-resilience-guide.md`
- 交付检查：`backend-checklist.md`

## 适用技术栈

- Java
- Spring Boot
- 常见服务端分层架构
- 数据访问、缓存、消息、任务流、配置管理和可观测性

## 基本要求

- 优先遵循项目现有架构和模块边界
- 接口契约、异常语义和状态码要清晰
- 输入校验尽量前置，减少可预防的 400 和模糊失败
- 事务边界、幂等性、一致性和兼容性要显式考虑
- 数据变更必须考虑迁移、回滚和历史数据影响
- 日志、监控、调试能力不能被忽略
- 测试不只看 happy path，还要覆盖失败路径和边界条件

## 架构与分层原则

- Controller 负责协议适配，不负责真正业务决策
- Service 承担业务编排和规则落地
- Repository 或 DAO 负责持久化，不承接业务含义
- 领域概念、错误语义、外部依赖边界要在代码中看得出来

## 可靠性与可观测性

- 关键链路要有足够日志上下文
- 能预见失败的位置要有明确错误处理策略
- 涉及外部调用时，考虑超时、重试、降级和幂等
- 对关键依赖考虑熔断、隔离和限流，避免故障扩散
- 发布后要能定位问题，而不是只能猜

## 性能与查询能力

- 查询语句和 ORM 使用要关注索引命中、扫描范围和分页策略
- 任何列表或搜索接口都要警惕大结果集和慢查询
- 避免 N+1、重复查询、无边界聚合和不必要的字段加载
- 性能不是上线后再看，设计时就应识别热点路径

## Spring Boot 方向

- 保持 Controller、Service、Repository 等职责清晰
- 避免把业务规则散落到控制层或数据层
- 配置、事务、校验、异常处理尽量走统一机制

## 交付标准

- 改动符合项目现有后端栈
- 关键契约、数据和异常路径已说明
- 代码不仅能跑，还便于维护和排障
