# TypeScript Guide

## 基本原则

- 用类型表达边界、契约和意图
- 类型应帮助收敛问题，不应成为噪音
- 优先定义稳定的数据结构，再写消费代码

## 推荐做法

- 为接口响应、表单值、组件 props、领域实体定义清晰类型
- 通过联合类型、字面量类型、类型收窄表达状态变化
- 对外暴露的函数和组件 API 保持稳定和明确

## 不推荐做法

- `any` 到处扩散
- 连续断言把错误压过去
- 类型命名模糊，无法表达领域含义
- 数据实际可空但类型假装永远存在

## 交付标准

- 类型错误不是靠屏蔽，而是靠建模解决
- 类型足够表达业务规则和状态边界
- 修改功能时，类型系统能提供真实帮助
