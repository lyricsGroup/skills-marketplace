# UI Testing

用于在实现完成后检查 UI 质量，并把结果落成 `.md` 文档。重点是关键路径是否可用，视觉和交互是否存在明显缺陷。

## 工作方式

1. 确认测试范围。
   明确页面、设备、浏览器、关键流程和重点风险。
2. 按流程检查。
   优先测主流程，再测异常路径、边界状态和视觉一致性。
3. 分类问题。
   将问题区分为阻塞、一般缺陷、建议优化。
4. 组织内容并落文件。
   默认按 [references/ui-test-checklist.md](references/ui-test-checklist.md) 生成或更新测试文档。
5. 如果用户没指定文件名或路径。
   默认写入仓库根目录 `docs/`，文件名使用 `YYYY-MM-DD-<topic>-ui-test.md`。

## 输出要求

- 默认生成或更新 `.md` 文件，而不是只在对话里直接展示测试结论
- 写明测试范围和通过情况
- 写明复现步骤和实际表现
- 给出整体 UI 结论
- 完成后在对话里回复文件路径、是否新建或更新、通过状态和阻塞问题概况，默认不把整篇文档全文回贴到对话里

## 给维护者的提示

- 如果团队使用固定验收标准，把标准写进 `references/`
- 这个 skill 输出的是测试结论，不替代代码审核
