# Component Reuse Guide

## 目标

- 提高一致性
- 降低重复实现
- 保持业务表达清晰
- 控制组件耦合度

## 推荐分层

- 基础组件：按钮、输入框、弹窗、表格容器等稳定 UI 原子
- 组合组件：搜索栏、筛选区、表单段落、列表卡片等可重复组合块
- 业务组件：带明确领域语义的组件，如订单状态卡、用户权限面板
- 页面组件：负责路由级编排，不承担底层复用职责

## 什么时候应该复用

- 相同交互语义反复出现
- 相同状态处理模式反复出现
- 多个页面需要共享统一视觉和行为

## 什么时候不应该复用

- 只有结构像，但业务含义和交互差异很大
- 为了“省文件数”把多个场景硬塞进一个组件
- 组件 API 已经复杂到调用方必须看源码才能用

## 判断标准

- 新增组件后，后续类似场景是否更容易实现
- 调用方是否更清晰，而不是更晦涩
- 组件边界是否稳定，未来不需要频繁破坏式修改
