# Vue Guide

## 使用原则

- 优先遵循 Composition API 和项目现有约定
- 组合逻辑抽离要服务于可读性和复用性，不是为了“更像框架专家”
- 模板层保持可读，不要把复杂逻辑塞进模板表达式

## 状态与组合逻辑

- 明确 props、emits、store、composable 的职责边界
- 对响应式数据的来源和写入位置保持清楚
- 避免多个 composable 共同操纵同一片状态却没有统一入口

## 组件设计

- 页面组件负责编排，基础组件负责稳定 UI 能力
- 业务组件优先围绕领域概念命名，而不是围绕 HTML 块命名
- 当模板开始难读时，应考虑拆分组件或下沉逻辑

## 常见问题

- 模板层条件分支过多
- setup 内部塞入所有逻辑但没有分层
- props / emits 不稳定，导致调用方心智负担过高
- 响应式便利性被滥用，最后很难追踪状态变化
